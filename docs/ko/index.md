## 왜 Nimbus 인가

또 다른 도구는 필요하지 않습니다. 당신에게 필요한 건 직원 — 반복 업무를 처리하고, 당신의 업무 방식을 학습하며, 새벽 2시에 고객 이메일이 와도 함께 깨어 있는 직원입니다. **Nimbus가 그 직원입니다.** 이미 사용 중인 앱에 연결하고, 당신이 직접 하던 단계를 그대로 따르며, 실행할 때마다 점점 더 똑똑해집니다.

| | |
|---|---|
| **자는 동안에도 일합니다** | 매일 팔로업, 주간 리포트, 시간 단위 점검 같은 작업을 원하는 시각에 예약하세요. Nimbus가 실행하고 끝나면 알려드립니다. |
| **이미 비용을 지불하고 있는 앱을 활용합니다** | Notion, Gmail, CRM, 캘린더. Nimbus가 로그인해 직접 처리합니다 — 복사·붙여넣기도, 내보내기도 없습니다. |
| **위험한 작업은 실행 전에 확인합니다** | 이메일을 직접 보내지 않고 초안을 작성하고, 행을 삭제하기 전에 계획을 먼저 보여줍니다. 결정권은 항상 사용자에게 있습니다. |
| **문제가 생기면 스스로 복구합니다** | 비밀번호가 바뀌었거나, 서비스가 재시작되었거나, 앱이 업데이트되어도 Nimbus가 감지하고 해결합니다. 알림이 울리지 않습니다. |

## 시맨틱 게이트웨이 (mcp.json)

<p style="color: #a3a3a3; font-size: 0.95em; margin: 0 0 16px 0;">
  <em>궁금한 분들을 위해: Nimbus는 직원이 사용할 수 있는 모든 앱을 나열한 작은 설정 파일 위에서 동작합니다. 직접 손댈 일은 거의 없습니다 — Nimbus는 합리적인 기본값을 함께 제공합니다.</em>
</p>

Nimbus는 모든 stdio 및 HTTP 기반 MCP 서버를 단일 인터페이스로 통합합니다. AI 모델이 기능을 요청하면 Nimbus가 Qdrant에 저장된 벡터 임베딩을 사용해 적절한 도구를 동적으로 발견하고 실행합니다.

Nimbus 홈 디렉터리의 `mcp.json` 을 수정해 활성 서버를 구성하세요.

```json
{
  "mcpServers": {
    "playwright-mcp": {
      "url": "http://localhost:3100/mcp",
      "transport": "streamable-http"
    },
    "notion-mcp": {
      "url": "https://mcp.notion.com/mcp",
      "transport": "streamable-http",
      "auth": "oauth"
    },
    "linkedin-mcp": {
      "command": "uv",
      "args": ["run", "-m", "linkedin_mcp_server", "--transport", "stdio"]
    }
  }
}
```

게이트웨이는 모델에 네 가지 주요 엔드포인트를 노출합니다 (`nimbus-utility-mcp` 네임스페이스 아래).

1. **`list_servers`** — 구성된 모든 MCP 백엔드 서버를 연결 상태와 도구 수와 함께 나열합니다.
2. **`find_tools`** — 모든 서버를 시맨틱하게 검색합니다. 모델이 "browse linkedin"을 요청하면 관련 도구 스키마만 반환합니다.
3. **`execute_tool`** — 등록된 모든 서버의 모든 도구를 동적으로 실행합니다.
4. **`chain_tools`** — 여러 단계를 한 번의 API 호출로 순차적으로 체이닝합니다 (예: 검색 → 읽기 → 보내기).

## CLI 레퍼런스

<p style="color: #a3a3a3; font-size: 0.95em; margin: 0 0 16px 0;">
  <em>기술 사용자를 위해: Nimbus가 이해하는 모든 명령입니다. 대부분의 사용자는 <code>nimbus start</code> 와 <code>nimbus doctor</code> 만 필요합니다 — 나머지는 문제가 생겼을 때를 위한 명령입니다.</em>
</p>

`nimbus` CLI는 로컬 컨테이너 워크스페이스의 라이프사이클을 관리합니다.

| Command | Action |
|:---|:---|
| `nimbus start` | 서비스 부팅. 모든 Docker 컨테이너 (postgres, redis, qdrant, gateway)를 프로비저닝하고 시작합니다. |
| `nimbus start --build` | 리빌드 후 부팅. 현재 소스에서 Docker 이미지를 다시 빌드하고 서비스를 깨끗하게 부팅합니다. |
| `nimbus start --gateway` | 헤드리스 모드. 게이트웨이 컨테이너만 실행합니다 — CI나 원격 MCP 전용 사용 시 전체 스택을 건너뜁니다. |
| `nimbus start --fresh` | 데이터 볼륨 초기화. Postgres/Redis 자격 증명을 재생성하고 데이터 디렉터리를 다시 초기화합니다. |
| `nimbus stop` | 종료. 모든 컨테이너 서비스를 안전하게 종료하고 시스템 포트를 해제합니다. |
| `nimbus upgrade` | 업그레이드. 최신 바이너리를 가져오고 래퍼 설정을 업그레이드합니다. |
| `nimbus update` | CLI 업데이트. 게이트웨이 컨테이너 이미지를 새로 고칩니다 (`--gateway` 와 함께 사용). |
| `nimbus uninstall` | 제거. Nimbus CLI, 컨테이너, 설정을 모두 제거합니다. |
| `nimbus uninstall --keep-data` | 제거하면서 데이터베이스, 키, 설정은 보존합니다. |
| `nimbus uninstall --purge` | 확인 없이 모든 것을 제거합니다. |
| `nimbus doctor` | 진단. Docker, 버전 차이, 컨테이너 상태, 업데이트 신선도, 자격 증명 드리프트를 확인합니다. |
| `nimbus recover-pg-role` | 복구. `nimbus_postgres_data` 볼륨을 현재 `.env` 자격 증명으로 마이그레이션합니다. |
| `nimbus recover-redis-password` | 복구. Redis 볼륨 비밀번호를 현재 `.env` 와 일치하도록 회전합니다. |
| `nimbus dashboard install` | 선택 사양인 Agent Dashboard 동반자를 설치합니다. |
| `nimbus dashboard uninstall` | Dashboard 컨테이너와 서비스 블록을 제거합니다. |
| `nimbus dashboard status` | Dashboard 활성화 여부와 컨테이너 상태를 표시합니다. |
| `nimbus config <subcommand>` | `~/.nimbus/.env` 를 관리합니다 (list, get, set, unset). |
| `nimbus mcp <subcommand>` | `~/.nimbus/mcp.json` 을 관리합니다 (list, get, set, remove). |
| `nimbus chat "prompt"` | Dashboard에 프롬프트를 보내고 응답을 스트리밍합니다. |
| `nimbus env-init` | CLI가 관리하는 시크릿을 `~/.nimbus/.env` 에 채워 넣습니다 (멱등). |

전체 레퍼런스는 [`nimbus --help`](download.md) 또는 [Download 페이지](download.md) 의 플랫폼별 설치 안내를 참고하세요.
