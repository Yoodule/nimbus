为什么选择 Nimbus

你不需要再多一个工具。你需要的是一位员工 —— 一位能够处理重复工作、学习你的工作方式、并在凌晨两点客户发来邮件时及时响应的员工。Nimbus 就是那位员工。 它接入你已经在使用的应用,按你手动完成的步骤执行,每一次运行都变得更加智能。
在你休息时继续工作 随时安排任务 —— 每日跟进、每周报告、每小时检查。Nimbus 执行后通知你。
利用你已经付费的应用 Notion、Gmail、CRM、日历。Nimbus 登录并操作 —— 无需复制粘贴,无需导出。
危险操作前先确认 发送草稿而非直接发邮件,删除行前先展示计划。决定权始终在你手中。
出问题时自行恢复 密码轮换、服务重启、应用更新 —— Nimbus 会自动检测并修复,无需打扰你。

它能做什么

📨 发送跟进邮件

告诉 Nimbus:"给上周下载白皮书但没有预约通话的所有人发邮件"。它会起草邮件、等待你审批、然后发送。

📊 从数据生成报告

"把上个月成交的交易按行业分组,在共享的 Notion 页面上加一张图表"。第二天早上完成。无需再为电子表格苦恼。

🗓️ 整理你的收件箱

每天早上,Nimbus 为新邮件打标签、为明确的内容起草回复,其余的在 Slack 中标记出来。你看到的只是需要你处理的部分。

🔁 按计划执行工作流

"每周五下午 4 点把本周的主要成果发到 #team-channel,然后给我发一封摘要邮件"。Nimbus 会一直执行,无需维护。

🧠 记住你的工作方式

你使用 Nimbus 越久,它学习得越多。它会记住你总是归档哪些邮件、哪些报告重要、哪些步骤可以省略。

🛡️ 未获你批准绝不发送

Nimbus 先起草,你再审批。没有你的点击,它无法发送邮件、删除行或发布消息。每个操作都有完整的审计日志。

适合独自承担一切的创始人

不要再把周日浪费在跟进邮件和更新 CRM 上了。把重复工作交给 Nimbus,把时间花在你才能完成的事上。

适合需要管控的企业

语义网关 (mcp.json)

面向好奇者:Nimbus 基于一个小型配置文件运行,文件列出了你的员工可以使用的所有应用。你可能不需要修改它 —— Nimbus 自带合理的默认值。

Nimbus 将所有基于 stdio 和 HTTP 的 MCP 服务器统一在一个接口之后。当 AI 模型请求某项能力时,Nimbus 使用存储在 Qdrant 中的向量嵌入动态发现并执行相应的工具。

在你的 Nimbus 主目录中编辑 mcp.json 来配置活动服务器:

{
  "mcpServers": {
    "playwright-mcp": {
      "url": "http://localhost:3100/mcp",
      "transport": "streamable-http"
    },
    "notion-mcp": {
      "url": "https://mcp.notion.com/mcp",
      "transport": "streamable-http",
      "auth": "oauth"
    },
    "linkedin-mcp": {
      "command": "uv",
      "args": ["run", "-m", "linkedin_mcp_server", "--transport", "stdio"]
    }
  }
}

网关向模型公开四个主要端点(位于 nimbus-utility-mcp 命名空间下):

  1. list_servers —— 列出所有已配置的 MCP 后端服务器及其连接状态和工具数量。
  2. find_tools —— 对所有服务器进行语义搜索。模型请求"浏览 linkedin"时,仅返回相关工具的 schema。
  3. execute_tool —— 动态执行任何已注册服务器上的任何工具。
  4. chain_tools —— 在一次 API 调用中按顺序链接多个步骤(例如:搜索 → 读取 → 发送)。

CLI 参考

面向技术用户:Nimbus 理解的所有命令。大多数用户只需要 nimbus start 和 nimbus doctor —— 剩下的留到出现问题时再用。

nimbus CLI 控制本地容器化工作区的生命周期。

命令 作用
nimbus start 启动服务。配置并启动所有 Docker 容器(postgres、redis、qdrant、gateway)。
nimbus start --build 重新构建并启动。从当前源码重建 Docker 镜像,然后干净地启动服务。
nimbus start --gateway 无头模式。仅运行网关容器 —— 跳过完整堆栈,适用于 CI 或仅使用远程 MCP 的场景。
nimbus start --fresh 清除数据卷。重新生成 Postgres/Redis 凭据,并重置数据目录。
nimbus stop 停止。安全关闭所有容器化服务并释放系统端口。
nimbus upgrade 升级。获取最新二进制并升级包装器的配置。
nimbus update 更新 CLI。刷新网关容器的镜像(与 --gateway 配合使用)。
nimbus uninstall 卸载。删除 Nimbus CLI、容器和配置。
nimbus uninstall --keep-data 卸载但保留数据库、密钥和配置。
nimbus uninstall --purge 卸载并在不确认的情况下清除所有内容。
nimbus doctor 诊断。检查 Docker、版本差异、容器健康状况、更新新旧、凭据漂移。
nimbus recover-pg-role 修复。将 nimbus_postgres_data 卷迁移到当前 .env 的凭据。
nimbus recover-redis-password 修复。轮换 Redis 卷的密码以匹配当前 .env。
nimbus dashboard install 安装可选的 Agent Dashboard 配套组件。
nimbus dashboard uninstall 移除 Dashboard 容器及其服务块。
nimbus dashboard status 显示 Dashboard 是否启用及其容器状态。
nimbus config <子命令> 管理 ~/.nimbus/.env(list、get、set、unset)。
nimbus mcp <子命令> 管理 ~/.nimbus/mcp.json(list、get、set、remove)。
nimbus chat "提示词" 向 dashboard 发送提示词并以流式获取响应。
nimbus env-init 将 CLI 管理的密钥写入 ~/.nimbus/.env(幂等)。

完整参考请参阅 nimbus --help,各平台安装步骤请参阅 下载页面。
